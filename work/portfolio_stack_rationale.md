# Week 4 — Stack Choice Rationale

## Chosen stack

I chose GitHub Pages as the stack for this portfolio build.

## Why this fits

This choice is the best match for my real constraints:

- It is free.
- It is simple to maintain.
- It is enough to display my work clearly.
- It does not require a backend right now.

## Three options considered

### 1. GitHub Pages

- Build: plain HTML/CSS
- Host: GitHub Pages
- Backend: none
- Trade-off: very lightweight and easy to keep running, but limited if dynamic features are needed later

### 2. Netlify

- Build: static site with simple hosting
- Host: Netlify
- Backend: none
- Trade-off: easy to publish and convenient, but more hosting complexity than I need for this stage

### 3. Vercel

- Build: small frontend app or static site
- Host: Vercel
- Backend: optional
- Trade-off: more powerful and polished, but heavier than necessary for a simple portfolio page

## Why I did not choose the others

I did not choose Netlify because it is useful, but it adds more hosting decisions than I need right now. I did not choose Vercel because it is more capable, but it is more than I need for a simple static portfolio page.

## Maintenance check

Can I maintain this? Yes. I can update a simple static page in the repo, push changes, and keep the site live with very little infrastructure overhead.

## Backend question

No backend is needed yet. This project is a static portfolio/publishing task, not a dynamic application, so the honest answer is that I do not need one at this stage.

## Final rationale

The main decision was not which tool is most impressive, but which one best fits my needs and can be maintained. GitHub Pages is the most honest and practical choice because it is free, reliable, easy to maintain, and good enough to show my work clearly.
